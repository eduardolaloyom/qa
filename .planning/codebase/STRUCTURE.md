## Directory Purposes

**ai-specs/**
- Purpose: Standardized AI workflow framework (reusable across projects)
- Contains: Role definitions, workflow orchestration templates, QA/testing standards
- Key files: `.agents/`, `.commands/`, `specs/`, `README.md`

**tests/e2e/**
- Purpose: Playwright E2E testing for web platforms (B2B tienda + Admin)
- Contains: 13 B2B specs (3200+ LOC total), 2 Admin specs, auto-generated fixture, helpers
- Key files: `b2b/*.spec.ts`, `fixtures/clients.ts` (AUTO-GENERATED), `playwright.config.ts`
- Entry point: `npm test` or `npx playwright test --project=b2b`

**tests/app/flows/**
- Purpose: Maestro YAML flows for mobile APP testing
- Contains: 13 generic flows + 11 client-specific (Prinorte) flows, master session composition
- Key files: `01-login.yaml` through `13-tareas-growth.yaml`, `helpers/`, `prinorte/`, `prinorte-session.yaml`
- Entry point: `maestro test tests/app/flows/`

**data/**
- Purpose: MongoDB extraction → QA matrix → test fixtures
- Contains: `mongo-extractor.py` (connection + parsing), 3 matrix JSON files (qa-matrix-staging.json is primary), feature status
- Workflow: Extract with `python3 data/mongo-extractor.py`, sync with `python3 tools/sync-clients.py`

**tools/**
- Purpose: QA automation and reporting scripts
- Contains: `sync-clients.py` (fixture generator), `run-qa.sh` (CLI wrapper), `live-reporter.js` (dashboard integration)
- Usage: Part of CI/CD pipeline and manual QA workflows

**checklists/**
- Purpose: Manual test cases by category (regression, tech debt, integrations, features)
- Contains: 8 categories × 5-10 checklists per category = 40+ markdown files
- Key files: `INDICE.md` (coverage map), `regresion/checklist-regresion-postmortems.md` (PM1-PM7)
- Linking: Each checklist item links to corresponding test (spec file, flow file, or manual step)

**QA/{CLIENT}/{DATE}/**
- Purpose: Results for a specific client on a specific date
- Contains: `cowork-session.md` (consolidated Cowork findings) + `qa-report-{DATE}.md` (generated report)
- Structure: One `cowork-session.md` per day (all modes A/B/C/D concatenated), one `qa-report-*.md` per execution

**public/**
- Purpose: GitHub Pages dashboard for QA results
- Contains: `index.html` (main dashboard), `manifest.json` (report index), `qa-reports/` (HTML reports)
- Workflow: `/report-qa` generates HTML report → added to manifest → dashboard auto-refreshes

**templates/**
- Purpose: Reusable templates for reports and communications
- Contains: `qa-report-template.md`, `cowork-qa-template.md`, `escalation-templates.md`
- Usage: `/report-qa` command fills these templates with client-specific data

## Naming Conventions

**Files:**
- Playwright specs: `{feature}.spec.ts` (e.g., `cart.spec.ts`, `prices.spec.ts`)
- Maestro flows: `{NN}-{feature}.yaml` where NN is two-digit sequential (e.g., `05-pedido.yaml`, `10-descuentos.yaml`)
- Checklists: `checklist-{category}-{area}.md` (e.g., `checklist-regresion-postmortems.md`)
- Reports: `qa-report-{YYYY-MM-DD}.md` or `{CLIENT}-{DATE}.html`
- QA directories: `QA/{CLIENT}/{YYYY-MM-DD}/` (e.g., `QA/Codelpa/2026-03-31/`)

**Directories:**
- Test fixtures: `tests/e2e/fixtures/`, `tests/app/config/`
- Client-specific flows: `tests/app/flows/{CLIENT}/` (e.g., `tests/app/flows/prinorte/`)
- Checklist categories: `checklists/{category}/` (regresion, deuda-tecnica, funcional, servicios, admin, integraciones)
- Results: `QA/{CLIENT}/{DATE}/` (flat two-level hierarchy)

**Variables & Constants:**
- Client credentials: `{CLIENT}_EMAIL`, `{CLIENT}_PASSWORD` (env var convention, hyphens → underscores)
- MongoDB flags: `camelCase` in `clients.ts` (e.g., `enableCoupons`, `purchaseOrderEnabled`, `hooks.stockHook`)
- Test IDs: `{CATEGORY}-{NN}` (e.g., `C1-01`, `C2-05`, `C3-E1`, `PM1`, `V1-03`)

## Where to Add New Code

**New Playwright Spec (B2B):**
- Primary code: `tests/e2e/b2b/{feature}.spec.ts` (follow multi-client pattern in `cart.spec.ts`)
- Update fixture if new flags needed: edit `data/b2b-feature-status.json` + re-run `python3 tools/sync-clients.py`
- Link in coverage: add entry to `checklists/INDICE.md` (which checklist does this test cover?)
- Pattern:
  ```typescript
  import { createClientTest, expect } from '../fixtures/multi-client-auth';
  import clients from '../fixtures/clients';
  
  for (const [key, client] of Object.entries(clients)) {
    const test = createClientTest(client);
    test.describe(`C{N} — Feature: ${client.name}`, () => {
      test(`${key}: C{N}-{NN} Test name`, async ({ authedPage: page }) => {
        // Use client.config.flagName to skip if not enabled
        if (!client.config.featureEnabled) {
          test.skip(true, 'Feature not enabled');
          return;
        }
        // ... test logic
      });
    });
  }
  ```

**New Maestro Flow (APP):**
- Primary code: `tests/app/flows/{NN}-{feature}.yaml` (use next available NN number)
- Client-specific: `tests/app/flows/{CLIENT}/{NN}-{feature}.yaml` (e.g., Prinorte-specific flows in `prinorte/`)
- Update master session if applicable: add `runFlow: {NN}-{feature}.yaml` to `prinorte-session.yaml` (or create new master)
- Link in coverage: update `checklists/INDICE.md` (which checklist maps to this flow?)
- Pattern (see `maestro-standards.mdc`):
  ```yaml
  appId: ${APP_PACKAGE}
  ---
  # {NN} - Feature Name
  # Tests: what this flow validates
  # Related: checklist ID or Linear ticket
  
  - runFlow: helpers/login.yaml
  - tapOn:
      id: "element_id"
  - assertVisible:
      id: "element_id"
  ```

**New Checklist (Manual Testing):**
- Location: `checklists/{category}/checklist-{category}-{area}.md` (pick existing category or create new)
- Structure: Sections for setup, test cases (ID, steps, expected), edge cases, known issues
- Link: Add entry to `checklists/INDICE.md` with reference to related test (spec, flow, or "Manual")
- Example: `checklist-regresion-postmortems.md` has PM1-PM7 linked to test files

**Utility Scripts:**
- Location: `tools/` (top-level for general utilities) or `tools/{category}/` (for domain-specific)
- Example: `tools/sync-clients.py` (cross-project), `tools/report/generate-grouped-report.py` (reporting)
- Pattern: Python for data processing, JS for Playwright plugins, Bash for CLI wrappers

**Configuration Changes:**
- Playwright: edit `tests/e2e/playwright.config.ts` (projects, reporters, timeouts)
- Maestro: edit `tests/app/config/config.yaml` or per-client `config.{CLIENT}.yaml`
- MongoDB: update `.env` with new URIs (MONGO_*_URI), re-run `mongo-extractor.py`

## Special Directories

**tests/e2e/fixtures/**
- Purpose: Shared fixtures, helpers, and auto-generated client config
- Generated: `clients.ts` is AUTO-GENERATED by `tools/sync-clients.py` — never edit manually
- Committed: Yes (generated files are committed to avoid build-time dependency)
- Update: After MongoDB config changes, run `python3 tools/sync-clients.py`

**tests/app/flows/helpers/**
- Purpose: Reusable Maestro subflows (login, sync, etc.)
- Committed: Yes
- Usage: Included via `runFlow: helpers/{name}.yaml` in other flows

**data/.app-mobile-cache/ and data/.b2b-source-cache/**
- Purpose: Cached source code snapshots for element inspection/debugging
- Generated: Yes (populated by extraction tools)
- Committed: No (in .gitignore) — auto-regenerated on demand

**public/**
- Purpose: GitHub Pages static assets (dashboard HTML, reports, manifest)
- Generated: Partially (HTML reports by `/report-qa`, manifest updated by report generator)
- Committed: Yes (manifest.json and historical reports committed; live.json excluded)

**QA/{CLIENT}/{DATE}/**
- Purpose: Results storage (not code)
- Generated: `qa-report-{DATE}.md` by `/report-qa` command
- Committed: Yes (results are part of project history)

**_archived/**
- Purpose: Deprecated files (kept for git history, not actively used)
- Committed: Yes
- Update: Move deprecated files here instead of deleting; link from README if needed

**.planning/codebase/**
- Purpose: Architecture and structure documentation (GSD mapping output)
- Files: ARCHITECTURE.md, STRUCTURE.md, CONVENTIONS.md, TESTING.md, STACK.md, INTEGRATIONS.md, CONCERNS.md (as applicable)
- Committed: Yes (reference for future Claude instances)
